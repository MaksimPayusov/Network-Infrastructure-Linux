# Сетевая инфраструктура из 3 виртуальных машин на Linux Ubuntu

> **Pet-проект**: Развертывание локальной сетевой инфраструктуры с сервисами DHCP, DNS, мониторингом и резервным копированием.

## Технологический стек

Ubuntu Server, Oracle VirtualBox, ISC Kea DHCP, BIND9, Prometheus, Grafana, Node Exporter, Bash, Netplan, OpenSSH, Git

## Описание проекта

Данный проект представляет собой учебную сетевую инфраструктуру, развернутую на 3 виртуальных машинах под управлением Ubuntu Live Server в среде Oracle VirtualBox.

### Основные сервисы

**ВМ1 — Шлюз и DHCP-сервер**
- Настроен Kea DHCP для динамической выдачи адресов в сети 10.0.0.0/24
- Включена маршрутизация (IP forwarding)
- Настроена интеграция с DNS-сервером через DDNS и TSIG-ключи
- Написаны скрипты автоматизации: backup.sh и health.sh

**ВМ2 — DNS-сервер и мониторинг**
- BIND9 настроен как мастер-сервер для зон lab.local и 0.0.10.in-addr.arpa
- Поддерживает динамические обновления от DHCP-сервера
- Развернуты Prometheus и Grafana для сбора и визуализации метрик
- На всех узлах работает Node Exporter для экспорта системных показателей

**ВМ3 — Клиентская машина**
- Получает IP-адрес автоматически через DHCP от ВМ1
- Резолвит имена через внутренний DNS-сервер ВМ2
- Участвует в мониторинге

## Структура репозитория

**configs/** — конфигурационные файлы, сгруппированные по виртуальным машинам

- `configs/vm1/` — настройки для шлюза и DHCP-сервера
  - `kea-dhcp4.conf` — основная конфигурация DHCP-сервера Kea
  - `kea-dhcp-ddns.conf` — параметры интеграции с BIND9 для DDNS
  - `netplan.yaml` — настройка сетевых интерфейсов
  - `sysctl.conf` — включение пересылки IP-пакетов
  - `node_exporter.service` — systemd-юнит для экспорта метрик

- `configs/vm2/` — настройки для DNS-сервера и системы мониторинга
  - `db.lab.local` — зональная файл для прямого разрешения имен
  - `db.10.0.0` — зональный файл для обратного разрешения (PTR-записи)
  - `named.conf.local` — конфигурация зон с политикой обновлений
  - `named.conf.options` — глобальные параметры BIND9
  - `netplan.yaml` — сетевая конфигурация
  - `prometheus.yml` — правила сбора метрик
  - `prometheus.service`, `grafana.ini` — настройки сервисов мониторинга
  - `node_exporter.service` — экспорт метрик с ВМ2

- `configs/vm3/` — настройки клиентской машины
  - `netplan.yaml` — интерфейс enp0s9 настроен как DHCP-клиент
  - `node_exporter.service` — экспорт метрик для мониторинга

**scripts/** — скрипты

- `backup.sh` — скрипт резервного копирования конфигурационных файлов с ВМ1
- `health.sh` — скрипт комплексной проверки работоспособности сети

## Архитектура сети

### Виртуальные машины и их назначение

**ВМ1 — Шлюз и DHCP-сервер**
- Статический адрес во внутренней сети: 10.0.0.1/24
- Адрес для управления с хоста: 192.168.56.10/24
- ОС: Ubuntu 24.04.4 Live Server

**ВМ2 — DNS-сервер и мониторинг**
- Статический адрес во внутренней сети: 10.0.0.2/24
- Адрес для управления с хоста: 192.168.56.11/24
- ОС: Ubuntu 24.04.4 Live Server

**ВМ3 — Клиентская машина**
- Получает адрес из пула 10.0.0.100–200 через DHCP
- Адрес для управления с хоста: 192.168.56.12/24
- ОС: Ubuntu 24.04.5 Live Server

### Сетевые интерфейсы

Каждая виртуальная машина оснащена тремя сетевыми адаптерами:

**Адаптер 1 — NAT (enp0s3)**
- Предоставляет доступ в интернет для установки обновлений и пакетов
- Все ВМ получают адреса в подсети 10.0.2.0/24 автоматически

**Адаптер 2 — Host-Only (enp0s8)**
- Изолированная сеть для управления ВМ с хостовой системы
- Статические адреса: 192.168.56.10–12/24
- Через этот интерфейс осуществляется SSH-доступ с пробросом портов

**Адаптер 3 — Internal Network "lab_net" (enp0s9)**
- Приватная сеть для взаимодействия сервисов между ВМ
- Подсеть: 10.0.0.0/24
- ВМ1 и ВМ2 имеют статические адреса, ВМ3 получает адрес через DHCP

### Доступ по SSH с хостовой системы (Windows)

Подключение к виртуальным машинам осуществляется через проброс портов

## Ключевые особенности конфигурации

### Интеграция DHCP и DNS через DDNS

Kea DHCP на ВМ1 настроен на отправку обновлений в BIND9 на ВМ2 с использованием ключа аутентификации TSIG. Алгоритм: HMAC-SHA256. BIND9, в свою очередь, разрешает обновление только записей типа A и PTR в пределах доверенных зон. Благодаря этому клиенты, получившие адрес по DHCP, автоматически регистрируют свое имя в зоне lab.local и обратную PTR-запись.

### Система мониторинга

Prometheus опрашивает экспортеры метрик на всех узлах с интервалом 15 секунд. Grafana предоставляет веб-интерфейс для визуализации данных по адресу `http://192.168.56.11:3000` и `http://10.0.0.2:3000`.

Цели для сбора метрик:
- `vm1-gateway` — метрики с шлюза (10.0.0.1:9100)
- `vm2-services` — метрики с сервера мониторинга и сам Prometheus (10.0.0.2:9100 и :9090)
- `vm3-client` — метрики с клиентской машины (динамический адрес в диапазоне 10.0.0.100–200)

### Скрипт резервного копирования (backup.sh)

Скрипт, размещенный на ВМ1, выполняет следующие действия:
- Создает архив конфигурационных директорий `/etc/kea` и `/etc/netplan`
- Сохраняет архив в `/backup/` с именем, содержащим дату и время
- Ведет лог выполнения в `/var/log/base_pet_project_backup.log`
- Автоматически удаляет архивы старше 7 дней

### Скрипт проверки здоровья сети (health.sh)

Универсальный скрипт, который можно запускать на любой ВМ. Проверяет:
- Связность между узлами через ping
- Корректность прямого и обратного разрешения имен через dig
- Доступность порта 53 на DNS-сервере
- Корректность резолвинга собственного имени хоста
